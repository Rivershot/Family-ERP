## 2026-05-27

### 1. Tailwind는 왜 클래스를 일일이 쓰나 (utility-first 철학)
- "클래스 정의 후 이름으로 적용"하는 방식(BEM, WebSquare 등)이 **전통적인 주류** 방식. Tailwind는 이를 의도적으로 뒤집은 **유틸리티-first**.
- 유틸리티 조합이 길어 보이는 건 정상이며 의도된 설계. 장점은 별도 CSS 파일 없이 마크업만 보고 스타일 파악 가능.
- **재사용은 CSS 클래스가 아니라 컴포넌트로** 묶는 게 React식 사고. (`@apply`는 알아두되 남용 금지 — 버튼·인풋 같은 진짜 반복 요소에만)

### 2. Flexbox 정렬
- flex 아이템은 **교차축으로 기본 stretch**됨 → `self-start`(`align-self: flex-start`)로 끊으면 내용 크기만큼만 차지.
- 주축 기본은 `justify-content: flex-start`(왼쪽 쏠림) → `justify-center`로 중앙 정렬.
- "빈 flex 공간 없이 내용 크기만큼"은 폭 강제(`w-full`/고정폭/`flex-1`)를 빼고 `w-fit`으로.

### 3. border-bottom (Tailwind 표기)
- `border-b`(1px), `border-b-2`, 임의값 `border-b-[3px]`. **`border-b-1`은 없는 클래스** (무시됨).
- 색은 별도: `border-violet-700`.
- **투명 보더 트릭**: 평소 `border-b-2 border-transparent`, 선택 시 색 지정 → 밑줄 생겨도 글자가 위아래로 안 밀림.

### 4. 그라데이션 텍스트
- 원리: 글자에 그라데이션 배경을 깔고 글자 모양으로 잘라냄 → `bg-clip-text text-transparent`.
- Tailwind v4는 `bg-linear-to-br from-... to-...` (v3는 `bg-gradient-to-br`). 135deg = `to-br`(좌상단→우하단).

### 5. 탭(Tab) 기초 패턴
- 핵심은 **"지금 어떤 탭이 활성인지"를 기억하는 state** 한 줄 (`useState`).
- 배열을 `map`으로 버튼 렌더 → `onClick`에서 `setState` → 조건부 렌더링으로 내용 전환.
- **라우트 기반 탭**: 활성 여부를 `useLocation`(현재 URL)으로 판단, 내용은 `Outlet`이 담당. 열린 탭 목록만 state로 관리.

### 6. inline style 작성법
- `style={{ ... }}` — **바깥 `{}`** = JSX 표현식 시작, **안쪽 `{}`** = JS 객체.
- 속성은 객체라 **쉼표로 구분**, 이름은 **camelCase** (`fontWeight`, `backgroundColor`).
- 길면 변수로 빼서 `style={tabStyle}`. (Tailwind 프로젝트에선 가능하면 className 우선, inline은 동적 계산값에)

### 7. 구조용 시맨틱 태그
- `aside`/`section`/`header`/`nav`/`main`/`footer` 등은 **표준 내장 태그**(커스텀 아님).
- 화면엔 `div`처럼 블록 박스로 보이고, **차이는 의미(접근성·가독성)뿐** — 기능/동작 없음.
- 반면 `a`(앵커)/`button`/`input`은 **실제 동작을 가진 태그**라 별개 분류.
- 한 줄 요약: *"구조용 시맨틱 태그 = 보기엔 div, 의미만 다른 표준 태그 / 앵커·버튼 = 진짜 동작하는 태그."*

### 8. Tailwind border-radius (`rounded-`)
- 스케일: `rounded-none`(0) / `xs`(2px) / `sm`(4) / `md`(6) / `lg`(8) / `xl`(12) / `2xl`(16) / `3xl`(24) / `4xl`(32) / `full`(완전 둥글게)
- 임의값: `rounded-[25px]` (스케일에 없는 값)
- **부분만 둥글게**: 변 단위 `rounded-t/b/l/r-*`, 모서리 단위 `rounded-tl/tr/br/bl-*` (예: `border-radius: 0 5px 5px 0` → `rounded-r-[5px]`)
- ⚠️ v4에서 이름이 한 칸 밀림: v3 `rounded`(4px) → v4 `rounded-sm`, v3 `rounded-sm`(2px) → v4 `rounded-xs`. v4엔 접미사 없는 `rounded` 안 씀.

## 2026-05-31

### 1. import 두 종류 — 중괄호 유무가 핵심
- **default import** (`import X from "..."`, 중괄호 없음): 이름을 **내 맘대로** 지어도 됨. 내보내는 쪽이 `export default 1개`라서 받는 쪽이 이름을 새로 붙이는 것. 한 파일당 1개만.
  - 예: `echarts-for-react`는 default export → `ReactECharts`든 `EChartsReact`든 자유. **단, 짓는 곳과 쓰는 곳 이름이 같아야** 함 (다르면 "정의 안 됨" 에러).
- **named import** (`import { X } from "..."`, 중괄호 있음): 정해진 이름표를 **그대로** 써야 골라낼 수 있음 (`useState`를 `usestate`로 쓰면 못 찾음). 한 번에 여러 개 가능. `as`로 별명 가능하지만(`{ useState as useS }`) 관례상 원래 이름 그대로 씀.
- **왜 이런가:** 옛날 `<script src>`는 전역에 고정 이름(`echarts`)이 박혔지만, ES 모듈은 전역 오염을 없애려고 "받는 쪽이 무엇을 가져올지 명시"하는 철학. default의 이름 자유는 그 결과.
- **실전:** 자유라도 보통 **라이브러리 README 권장 이름** 사용 + **IDE 자동완성**이 import 문을 대신 적어줌 (제일 편함).

### 2. useState 작성·사용
- `useState`는 **초깃값을 인자로 받고**, `[현재값, setter함수]` **배열을 돌려줌**.
- 그래서 **구조분해**로 받는 게 표준: `const [data, setData] = useState(초깃값);`
- ❌ 흔한 실수: `const x = useState(["a","b"])` → 이건 `["a","b"]`를 *초깃값*으로 넣은 꼴 (이름 짓기가 아님). 반환값을 통째로 받아 배열 인덱스로 쓰게 되어 의도와 어긋남.
- 안 쓰는 state는 지워도 됨. 데이터를 props/state로 다룰 때 위 구조분해 패턴으로 다시 작성.

### 3. ECharts를 React에 붙이기 (echarts-for-react)
- 옛날 방식(`useRef`+`useEffect`로 DOM에 직접 mount) 대신 `<ReactECharts option={...} />` 한 줄로 끝.
- `option` 객체가 차트의 전부. **풀블리드(카드 끝까지 채우기)** = `grid:{left:0,right:0,top:0,bottom:0}` + 부모 카드 `overflow-hidden`.
- 곡선 영역 차트: `series.type:"line"` + `smooth:true` + `symbol:"none"`(점 숨김) + `areaStyle`(그라데이션). 축은 `show:false`로 숨김.
- 그라데이션은 `echarts.graphic.LinearGradient` import 없이 **plain object**(`{type:"linear", x/y/x2/y2, colorStops:[...]}`)로도 가능 → 더 간단.
- ⚠️ 데이터(`series.data`)가 없으면 안 그려짐 → 처음엔 가짜 숫자 배열로 시작, 실데이터는 나중에 props로.

### 4. TypeScript에서 컴포넌트 props 작성법
- props의 **모양(타입)을 먼저 정의**하고, 함수 매개변수에 그 타입을 붙인다. (자바에서 메서드 파라미터 타입 선언하던 것과 같은 개념 — "이 자리엔 이 타입만")
  ```tsx
  type BudgetChartProps = {
    data: number[];
    color?: string;              // ? = optional, 안 넘기면 undefined (없으면 필수)
    kind: "expense" | "budget";  // 리터럴 유니온 = 정해진 값만 허용
  };
  function BudgetChart({ data, kind }: BudgetChartProps) { ... }  // 구조분해 + : 타입
  ```
- **핵심은 `{ 구조분해 }: ~Props`** — 이게 있어야 TS가 타입을 알고 잘못 쓰면 빨간줄.
- **기본값**: 구조분해에서 `=` → `function BudgetChart({ color = "#3B5BFE" }: Props)` (optional `?`과 짝으로 자주 씀).
- `type` / `interface` 둘 다 되지만 컴포넌트 props엔 **`type`이 더 일반적**. props 타입은 보통 **컴포넌트 함수 바로 위**에 둠.
- `React.FC`는 옛날 방식 — 요즘은 위처럼 **매개변수에 직접 타입** 다는 걸 권장.
- 관련: [[props vs state]] — "밖에서 들어오면 props, 스스로 바꾸면 state".

### 5. ECharts xAxis는 type에 따라 data 의미가 다름
- **`type:"category"`** 일 때 → `xAxis.data`는 **분류(라벨) 데이터**. 즉 `["1월","2월","3월"...]` 같은 "각 점이 무엇/언제인지"를 뜻함.
- 이건 **세로 높이값(`series.data`)과 전혀 다른 것** — 가로축=언제/무엇(라벨), series.data=얼마(높이). **같은 배열을 양쪽에 넣으면 안 됨** (값 중복 시 칸 꼬임).
- category 축에 data를 안 주면 ECharts가 series 개수만큼 0,1,2... 자동 생성 → 축을 `show:false`로 숨긴 장식용 스파크라인은 **비워둬도 정상**.
- 축에 라벨을 실제로 보여줄 차트면 그때 `["1월",...]` 넣고, 길이는 `series.data` 개수와 일치시킴. 순번만 필요하면 `data.map((_, i) => i)`.

### 6. props vs state — 재사용 컴포넌트의 핵심 갈림길
- **들어오는 값이면 props, 스스로 바꾸는 값이면 state.**
  | | props | state (`useState`) |
  |---|---|---|
  | 누가 정하나 | **부모(밖)가** 넣어줌 | 컴포넌트 **자신이** 바꿈 |
  | 언제 | "이 카드는 지출용/예산용"처럼 밖에서 결정 | 토글·입력값처럼 스스로 변함 |
  | 비유 | 함수의 **인자** | 함수 안 **지역변수** |
- **재사용 컴포넌트는 props가 정답**: `<BudgetChart data={...} kind="budget" />` 처럼 부모가 차이값을 넣어줌 → 컴포넌트 하나로 여러 변형을 찍어냄("껍데기 1개 + 내용 N개").
- ⚠️ **안 변하는 설정을 state에 넣지 말 것**: state는 "변해서 화면을 다시 그려야 하는 값"에만. 차트 설정(type/color/smooth)처럼 고정된 건 그냥 변수나 props로.
- ⚠️ **`useState(초깃값)`의 초깃값은 맨 처음 1회만 읽힘**: 초깃값 안에 다른 state(`data: chartData`)를 박아두면 그 순간 값을 *복사해 얼려둠* → 원본이 바뀌어도 안 따라감(동기화 안 됨). 데이터가 두 군데로 쪼개지는 버그.
- 데이터가 API 등으로 갱신되면 → **데이터를 소유한 부모가 state로 들고**, 자식엔 props로 내려줌. 관련: [[ts-props 작성법]]

### 7. TypeScript 타입 추론 — "항상 적는 게 아니다"
- TS 기본은 **추론(inference)**: 값을 보고 타입을 자동으로 알아냄. 자바처럼 다 선언하는 게 아니라 **안 적는 게 정상**.
  ```tsx
  const n = 3;            // number
  const arr = [1,2,3];    // number[]
  const obj = {a:1,b:"x"} // { a:number; b:string }
  ```
- **타입을 직접 적어야 하는 경우만:**
  | 상황 | 이유 |
  |---|---|
  | 함수 매개변수 / 컴포넌트 props | TS가 못 알아냄 → **필수** |
  | 빈 배열·늦은 할당 (`useState([])`, `let x`) | `any[]`로 추론돼버림 |
  | 추론이 의도와 다를 때 | 추론(넓음) ≠ 원하는 타입(좁음) |
  | 계약을 못박고 싶을 때 | 오타·실수를 컴파일에서 잡으려고 |
- 한 줄: **"기본은 추론, 추론이 안 되거나 틀릴 때만 명시."** [[ts-props 작성법]]도 매개변수라 명시가 필수인 케이스.

### 8. AG-Grid (v35) 사용 방식
- 버전마다 셋업이 크게 다름. **v33+는 옛날 튜토리얼대로 하면 안 됨** (CSS import·theme class 방식 폐기).
- **필수 3가지:**
  1. **모듈 등록 (한 번)**: 없으면 런타임 에러 #200.
     ```tsx
     import { AllCommunityModule, ModuleRegistry } from "ag-grid-community";
     ModuleRegistry.registerModules([AllCommunityModule]);
     ```
  2. **`rowData`(배열) + `columnDefs`**: colDef의 `field`가 rowData 객체의 **key와 매칭**돼야 그 칸에 값이 들어감.
  3. **컨테이너에 명시적 높이** (ECharts처럼): `<div style={{height:250}}><AgGridReact .../></div>` — 높이 없으면 안 보임.
- ❌ **CSS import 안 함** (`ag-grid-community/styles/*.css` 경로 없어짐), ❌ **`className="ag-theme-*"` 안 붙임** → v33+는 **Theming API**가 기본(`themeQuartz` 자동 적용).
- **TS 타입**: `columnDefs`에 `field:"num"`을 그냥 쓰면 literal이 `string`으로 넓혀져 에러(`string` is not assignable to `NestedFieldPaths`). AG-Grid는 field를 rowData의 key로 좁게 요구하기 때문 → 행 타입을 정의해 `ColDef<RowType>[]`로 맞춰줌(오타도 잡아줌). 느슨하게는 `ColDef[]`. [[ts-타입추론]]의 "추론이 의도와 다를 때" 케이스.

### 9. AG-Grid 컬럼/정렬 옵션 + 내부 스타일링
- **컬럼 폭**: `flex`는 **상대 비율**(`flex:1` 균등). 한 컬럼만 좁히려면 그 colDef에 `flex:0.5`. 고정폭은 `maxWidth`(flex와 width는 같이 못 씀).
- **인터랙션 끄기**: `defaultColDef`에 `resizable:false`(폭 조절 핸들=헤더 세로선 제거), `sortable:false`(헤더 클릭 정렬). 보여주기용 그리드는 둘 다 끄면 깔끔.
- **데이터 셀 가운데**: colDef/`defaultColDef`에 `cellStyle:{ textAlign:"center" }` (셀은 일반 블록이라 text-align 먹음).
- **헤더 가운데 (별도 방식)**: 헤더는 `cellStyle` 같은 인라인 prop 없음 → **`headerClass`(클래스 부여) + 전역 CSS** 조합.
  ```tsx
  defaultColDef={{ ..., headerClass:"ag-header-center" }}
  ```
  ```css
  /* index.css */
  .ag-header-center .ag-header-cell-label { justify-content: center; }
  ```
- **⚠️ 핵심 깨달음 — AG-Grid 내부 구조를 알아야 함:**
  - AG-Grid가 렌더하는 요소(`.ag-header-cell-label` 등)는 **컴포넌트 바깥에서 생성**돼서 JSX에 Tailwind 유틸을 못 붙임 → **전역 CSS(index.css)에 내부 클래스를 타겟**해야 함.
  - 헤더 텍스트 컨테이너(`.ag-header-cell-label`)가 **flex container**라 `text-align:center` ❌ → **`justify-content:center`** 로 정렬. (데이터 셀과 정렬 방식이 다른 이유)
  - 즉 theme 파라미터로 안 되는 세밀한 커스터마이징은 **내부 DOM 클래스 구조를 알고 직접 CSS로 타겟**하는 게 AG-Grid의 흔한 패턴.

## 2026-06-01

### 1. 같은 이름의 type + function = "already declared"
- TS는 **타입 네임스페이스**와 **값 네임스페이스**가 따로라 보통 같은 이름의 타입·값이 공존 가능(예: `interface Foo` + `const Foo`).
- 단 `function` 선언은 **스코프 식별자 바인딩**이라, 같은 이름의 `type X` + `function X`는 **충돌**(`already declared`).
- 해결: 둘 중 하나 rename. **관례 = props 타입은 `컴포넌트명 + Props`** (예: `ExpenseChartProps`). 타입 이름과 `function f(...): 타입` 두 군데를 같이 맞춤.

### 2. 빈 배열 타입 주석 (ts7034 — implicit any)
- `let x = [];` → 원소가 없어 TS가 element 타입을 추론 못 함 → `any[]` → `strict`(noImplicitAny)에서 에러.
- `const arr = [1,2,3]`처럼 **원소가 있으면 추론 OK** (안 적어도 됨). 빈 배열만 못 함.
- 해결: 직접 타입 주석 → `let expenseData: number[] = [];` (받는 props 타입과 동일하게).

### 3. 타입 주석, 언제 적고 언제 생략하나 (판단 기준)
- TS 기본은 **추론** → 정보가 충분하면 **안 적는 게 관례**. (자바처럼 매번 적지 않음)
- 한 줄 기준: **"초기값이 앞으로 들어올 데이터를 대표하는가?"**
  - 대표함(`const n=100`, `[1,2,3]`) → 추론에 맡김.
  - 지금은 비었지만 나중에 채울 것(`[]`, `null`, 미대입 `let x`) → **직접 주석**.
- 항상 적는 자리: **함수 매개변수 / 컴포넌트 props**(호출 전엔 알 수 없음), 추론(넓음)이 의도(좁음)와 다를 때. [[ts-타입추론]] 보강.

### 4. import 대소문자 에러 (TS1261) + macOS 파일시스템
- 증상: `tsc --noEmit`는 통과하는데 **에디터에만** `differs ... only in casing` 빨간 줄.
- 원인: macOS/Windows FS는 **대소문자 구분 안 함**(`Expense`=`expense` 같은 파일). 파일을 소문자로 만들었다 대문자로 rename하면 FS는 넘어가도 **TS 서버가 옛 이름을 캐시**해 한 파일을 두 이름으로 인식 → 충돌.
- 즉 코드/디스크 문제 아닌 **TS 서버 stale 캐시**. 해결: VS Code `Cmd+Shift+P` → **"TypeScript: Restart TS Server"** (또는 Reload Window).
- ⚠️ 교훈: **대소문자만 바꾸는 rename은 git/TS가 놓치기 쉬움.** Linux 배포 서버는 대소문자를 **구분**하므로 import 경로 ≠ 실제 파일명이면 로컬은 멀쩡해도 배포 때 깨짐 → 처음부터 일치시킬 것.

### 5. 콜백 — "함수를 값으로 넘기기" (ECharts formatter / onClick 공통)
- `formatter`, `onClick` 같은 자리는 **함수를 값으로 넘기는 것**. 내가 부르는 게 아니라 **프레임워크가 정해진 시점에 대신 호출**(콜백)하고, **return한 값**을 사용함. (Java 람다·이벤트 핸들러와 같은 개념)
- ⚠️ **괄호 유무가 핵심:**
  - `formatter: addComma` → 함수 **자체**를 넘김 ✅ (나중에 호출됨)
  - `formatter: addComma()` → **지금 즉시 실행한 결과**를 넘김 ❌ (의도 아님). `onClick={fn}` vs `onClick={fn()}`도 똑같은 함정.
- **화살표 본문 두 형태:**
  - `(p) => p.value.toLocaleString()` — 중괄호 X → **자동 return**(한 줄)
  - `(p) => { return ...; }` — 중괄호 O → `return` 직접 안 쓰면 `undefined` 반환(라벨 안 나옴)
- **인자**: 입력이 필요하면 `(p) => fn(p.value)`로 받아 넘김. `() => fn()`처럼 비우면 넘어온 데이터를 못 씀.
- **TS 타입**: ECharts `option`은 강타입이 아니라 콜백 `p`가 implicit any → 직접 명시. **쓸 필드만 좁게**: `(p: { value: number }) => fn(p.value)` (구조적 타이핑 — "이 모양만 있으면 OK"). 막대 차트 값은 숫자라 `value:number`로 충분. [[ts-타입추론]]의 "추론 안 되면 명시" 케이스.

### 6. ECharts 상태(state) — normal vs emphasis
- ECharts 요소는 **상태별로 스타일을 따로** 가짐. "평소 모습"과 "강조됐을 때 모습"을 분리 정의.
  | 상태 | 언제 | 위치 |
  |---|---|---|
  | **normal** | 평소(기본) | 옵션 **맨 위 레벨** (`label`, `itemStyle`, `labelLine`…) |
  | **emphasis** | **강조될 때** (기본 트리거 = **마우스오버**) | `emphasis: { ... }` 안에 중첩 |
- emphasis = "강조 상태". hover가 대표 예시지만 그것만은 아님 — **범례 hover, 코드 `dispatchAction` 하이라이트**로도 emphasis가 됨.
- **함정**: normal에서 `label:{show:false}` 껐어도 **emphasis는 별도**라 hover 시 다시 나타남 → `emphasis.label.show:false`도 따로 꺼야 함. labelLine(지시선)도 동일하게 양쪽.
  - ⚠️ `emphasis: { lable: {...} }` 처럼 **오타 내도 TS가 못 잡음** (option이 느슨한 타입이라 모르는 키 통과) → 조용히 무시됨. 안 먹히면 키 철자부터 확인.
- 다른 상태도 있음: **`blur`**(딴 걸 강조할 때 나머지 흐려짐), **`select`**(클릭 선택). 전부 같은 "상태별 스타일" 패턴.

### 7. 차트 label vs tooltip — "항상 보이면 label, 올려야 뜨면 tooltip"
- **label**: 차트 위/옆에 **상시 그려지는** 글자 (조각·막대 라벨). 끄기 = `label:{show:false}` (+ 상태별로 emphasis도). 조각-라벨 잇는 직선 = **`labelLine`**.
- **tooltip**: 마우스 올리면 **떠오르는 말풍선**. 끄기 = `tooltip:{show:false}`. ECharts는 tooltip 설정 안 하면 원래 안 뜸(뜨면 어딘가 `tooltip:{...}`가 있는 것).

### 8. 라이브러리 콜백 타입은 "직접 만들지 말고 찾아 쓴다" + 옵셔널 체이닝
- 콜백 param 타입은 **라이브러리가 export해둔 걸 쓰는 게 정석**. 손으로 모양 적는 건 그게 없을 때만.
  - **ECharts** formatter → option이 느슨한 타입이라 param 타입 안 줌 → 어쩔 수 없이 직접 `(p: { value: number })`.
  - **AG-Grid** → `ICellRendererParams<TData>` **제공** → `import type { ICellRendererParams } from "ag-grid-community"` 후 `(p: ICellRendererParams<Family>)`. 이미 `ColDef<Family>`로 행 타입 선언했으니 `Family`만 끼우면 `p.data`가 통째로 타입됨(필드별로 안 적어도 됨, 자동완성·오타체크 ✓).
- **제네릭 ≠ "항상 값 있음"**: `ICellRendererParams.data`는 정의가 `data: TData | undefined`라 `<Family>` 넣어도 **`Family | undefined`**.
  - **제네릭 = 모양**("있으면 Family"), **`| undefined` = 존재 여부**("없을 수도") — 별개의 질문이라 따로 표현됨.
  - 왜 undefined? AG-Grid가 **데이터 없는 행에도 렌더러 호출**(그룹 행·로딩/스켈레톤 행·디테일 행) → "항상 데이터 행이라고 약속 못 함" → 호출측이 방어하도록 강제(안전한 API 설계).
- **대응**: ① `p.data?.img` 옵셔널 체이닝(없으면 undefined, 안 터짐 — **AG-Grid 렌더러선 그냥 항상 이거 쓰는 게 편함**) ② `p.data!.img` 단언("없을 리 없다" 확신 시) ③ `if(!p.data) return null` 가드(이후 Family로 좁혀짐).
- `strictNullChecks`(strict)가 "undefined일 수 있는 값 까기"를 막는 거라 **에러가 아니라 정상 경고**. API 데이터(axios) 다룰 때 계속 만남 → `?.` 손에 익히기. [[ts-타입추론]] 연장.

### 1. 비동기 데이터 3박자 (axios + useState + useEffect)
- 하드코딩 `let x = [...]` 와 달리 axios는 값이 **즉시 안 옴(비동기)**. `const x = axios.get(...)` 하면 x엔 데이터가 아니라 **Promise(약속)** 가 담김 → 안 됨.
- 정석 패턴:
  1. **빈 그릇**: `const [x, setX] = useState<T[]>([])` — 처음엔 `[]`라 화면 일단 그려짐.
  2. **요청**: `useEffect(() => { ... }, [])` — 의존성 `[]` = "최초 1회만". (본문에서 직접 부르면 setState→리렌더→재요청 **무한루프**)
  3. **채우기**: `axios.get(url).then((res) => setX(res.data))` — 응답 오면 state 갱신 → React가 자동 리렌더.
- ⚠️ 데이터는 `res`가 아니라 **`res.data`** (axios가 `{data,status,headers...}`로 감쌈).
- 전통 웹 비유: 화면 빈 채로 먼저 그리고 `onload`/AJAX 콜백에서 `grid.setData(응답)` 하던 것. 차이 = React는 **state 바뀌면 알아서 다시 그림**.

### 2. public 폴더 경로 함정 (디스크 경로 ≠ URL)
- `public/`는 dev 서버가 **루트(`/`)로 서빙** → 폴더 이름 `public`이 **URL에 안 들어감**.
  - 파일 `public/mock/schedule.json` → URL **`/mock/schedule.json`**.
  - (이미 쓰던 예) `public/avatars/a.svg` → `/avatars/a.svg`.
- `/public/...` 로 요청하면 **404**.

### 3. JSON 최상위는 객체가 아니어도 됨 + key 유무
- JSON 파일 전체가 **배열/숫자/문자열**이어도 valid. `expense.json` = `[18400, 14000, ...]` (key 없이 숫자 나열).
- key는 **항목이 객체일 때만** 필요 — `schedule`은 객체 배열이라 `"id"`/`"date"` 등 key 필요, `expense`는 숫자라 이름 붙일 게 없음.
- JSON 빡빡한 문법: **모든 key 큰따옴표**, 문자열도 **큰따옴표**, **trailing comma 금지**, 숫자는 따옴표 없이.

### 4. axios 응답 타입 — `res.data`는 기본 `any`
- axios `AxiosResponse<T=any>` → **`res.data`는 `any`**. 통째로 `setX(res.data)` 넘길 땐 문제 없지만, **`res.data.map(...)`** 하면 `any`에 `.map`이라 콜백 `val`/`idx`가 **implicit any 에러(ts7006)**.
  - 이유: `any`에 메서드 호출하면 콜백 매개변수에 **contextual type이 안 생김** → `noImplicitAny`가 잡음.
- 해결 = **출처에서 타입 박기**: `axios.get<Record<string, number>[]>(url)` → `res.data`가 그 타입이 되고 `.map`의 `val`이 자동 추론 → 에러 사라짐. (인라인 `(val: T) =>`도 되지만 출처 박는 게 깔끔)
- `Record<string, number>` = `{ "문자열key": 숫자 }` 모양.

### 5. Object.keys vs Object.values — key·value 헷갈림 주의
- `{ "Income": 6500000 }` 에서 **이름="Income"=key**, **숫자=6500000=value**.
  | 원하는 것 | 메서드 | 결과 타입 |
  |---|---|---|
  | 이름 | `Object.keys(val)[0]` | `string` |
  | 숫자 | `Object.values(val)[0]` | `number` |
- 실수 사례: 이름 뽑으려고 `Object.values(val)[1]` 씀 → ① values는 숫자라 `number[]`인데 `string[]`에 대입 → **ts2322**(number→string 불가) ② key 하나뿐이라 `[1]`은 `undefined`. → `Object.keys(val)[0]`로 교정.
- 같은 결의 함정: `val[idx]` 로 객체 값 못 꺼냄 — `val[0]`은 "0이라는 key"를 찾는데 key는 `"Income"` → undefined. 객체 값은 `Object.values`로.
