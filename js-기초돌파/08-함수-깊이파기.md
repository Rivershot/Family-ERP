# 08. ⛏️ 함수 깊이 파기 — 선언/표현식/화살표, `this`, `call`/`apply`/`bind`, 클로저

> **목표:** 화살표 함수의 `this`가 왜 다른지, 클로저가 무엇인지(=`useState`·핸들러의 정체)를 안다.

## 왜 필요한가
콜백(`map`/`filter` 인자), 화살표 함수, `useState`, 이벤트 핸들러 — 전부 함수다.
특히 **클로저**는 React 훅이 동작하는 원리고, **`this`**는 클래스([[10-클래스]])·레거시 코드 읽기에 필수.

---

## 1) 만드는 3가지 방법
```js
function decl(x) { return x; }          // ① 선언문 — 호이스팅됨(위에서 호출 가능)
const expr = function (x) { return x; } // ② 표현식 — 변수라 선언 후에만
const arrow = (x) => x;                 // ③ 화살표 — 간결 + this 안 가짐(아래 3번)
```
- **선언문**은 [[06-변수와-스코프]] 호이스팅으로 정의 위에서도 호출 가능. **표현식/화살표**는 변수라 TDZ → 선언 후에야.

## 2) 화살표 함수 — 짧게 + `return` 규칙
```js
(n) => n * 2;              // 한 줄 → 자동 return (01강에서 봄)
(n) => { return n * 2; };  // 중괄호 → return 직접
() => ({ a: 1 });          // 객체 반환은 () 로 감싸기 (안 그럼 {}를 블록으로 오해)
```

## 3) `this` — 일반 함수 vs 화살표 ⭐
`this`는 **"누가 불렀냐(호출 방식)"** 로 정해진다(선언 위치 아님).
```js
const obj = {
  name: "채원",
  hi() { return this.name; },           // 일반 메서드: obj가 부름 → this=obj
};
obj.hi(); // "채원"

const fn = obj.hi;
fn();     // undefined ← 그냥 호출되면 this를 잃음(전역/undefined)
```
**화살표는 자기 `this`가 없다 → 바깥(정의된 곳)의 `this`를 그대로 씀:**
```js
const obj2 = {
  name: "민지",
  items: [1, 2],
  run() {
    this.items.forEach((n) => console.log(this.name, n)); // 화살표라 this=run의 this=obj2 ✅
    // function 콜백이었으면 this를 잃어 undefined.name 터짐
  },
};
```
→ React에서 콜백/핸들러를 화살표로 쓰는 이유: **바깥 컴포넌트의 값을 그대로** 쓰려고.

## 4) `call` / `apply` / `bind` — `this`를 강제 지정
```js
function intro() { return `${this.name} 입니다`; }
const a = { name: "채원" };

intro.call(a);          // "채원 입니다"  (this=a로 즉시 호출)
intro.apply(a);         // 같음. 인자를 배열로 넘김: fn.apply(a, [1,2])
const bound = intro.bind(a);  // this=a로 고정된 새 함수 반환(호출은 나중)
bound();                // "채원 입니다"
```
- `call`/`apply` = **즉시 호출**(인자 전달 방식만 다름: 낱개 vs 배열).
- `bind` = **this 박은 새 함수**를 만들어 둠(이벤트 핸들러 등록 등). *레거시 클래스 컴포넌트에서 `this.handleX = this.handleX.bind(this)` 가 이것 — 요즘 화살표로 대체.*

## 5) 클로저 — "함수가 태어난 환경(변수)을 기억" ⭐⭐
```js
function makeCounter() {
  let count = 0;                 // 바깥 변수
  return () => {                 // 안쪽 함수가 count를 "기억"
    count += 1;
    return count;
  };
}
const next = makeCounter();
next(); // 1
next(); // 2   ← count가 살아남아 누적 (makeCounter는 이미 끝났는데도!)
```
- **클로저** = 함수 + 그 함수가 **만들어질 때 보던 바깥 변수들**을 묶어 기억하는 것.
- `count`는 밖에서 못 건드리고 `next`를 통해서만 바뀜 → **데이터 은닉**.
- ⭐ **`useState`가 이 원리.** 리렌더돼도 값이 유지되고, 이벤트 핸들러는 자기가 만들어질 때의 state·props를 기억한다(=클로저).

## 6) 고차함수(HOF) — 함수를 주고받는 함수
```js
arr.map((n) => n * 2);    // map은 "함수를 인자로 받는" 고차함수
function once(fn) {        // 함수를 받아 함수를 반환
  let done = false;
  return (...args) => { if (!done) { done = true; return fn(...args); } };
}
```
- `map`/`filter`/`reduce`가 전부 고차함수. 너는 이미 매일 쓰고 있었다.

---

## ⛏️ 더 파기 (한 바퀴 돈 뒤 / 막힐 때)
- **`this` 4가지 규칙:** ① `new`(인스턴스) ② 명시 바인딩(`call`/`bind`) ③ 메서드 호출(점 앞 객체) ④ 그냥 호출(전역/undefined). 우선순위 ①>②>③>④. 화살표는 이 규칙을 **건너뛰고** 바깥 this.
- **stale closure(낡은 클로저) 함정:** 이벤트 핸들러/`setTimeout`이 **옛 state를 기억**해 버그. (예: `setCount(count+1)` 여러 번 → 같은 count 봄) → 해법 `setCount(prev => prev+1)`(함수형 업데이트). React 02강([[react-실전]])에서 다시 만남.
- **클로저와 메모리:** 클로저가 큰 객체를 붙잡으면 GC가 못 치움(누수 가능).

## ✅ 체크포인트
- `this`는 **호출 방식**으로 결정. 화살표는 자기 this 없이 **바깥 this** 사용.
- `call`/`apply`=즉시 호출(this 지정), `bind`=this 고정한 **새 함수**.
- **클로저** = 함수가 태어난 환경의 변수를 기억 → `useState`·핸들러의 원리.
- `map`/`filter`/`reduce` = 함수를 받는 **고차함수**.
