# 03. 배열 탐색·정렬 — find / findIndex / some / every / includes / sort

> **목표:** "찾기/있나없나/정렬"을 메서드로 처리한다. for 돌며 플래그 변수 쓰던 습관 버리기.

## 왜 필요한가
- 카테고리에 맞는 색 찾기, 특정 멤버 찾기 = **탐색**.
- "결제 대기 건이 하나라도 있나?" = `some`. "전부 승인됐나?" = `every`.
- 일정/지출을 날짜순·금액순 보여주기 = `sort`.

```js
const members = [
  { id: 1, name: "채원", age: 34, role: "admin" },
  { id: 2, name: "민지", age: 31, role: "user" },
  { id: 3, name: "지호", age: 7,  role: "user" },
];
```

---

## 1) `find` — 조건 맞는 **첫 원소** 하나
```js
members.find((m) => m.role === "admin"); // { id:1, name:"채원", ... }
members.find((m) => m.age > 100);        // 못 찾으면 undefined
```
- filter는 **배열**(여러 개), find는 **원소 하나**(첫 번째). 하나만 필요하면 find.
- `findIndex` — 위치(인덱스)가 필요하면. 없으면 `-1`.

## 2) `some` / `every` — **true/false 한 개**
```js
members.some((m) => m.role === "admin");  // 하나라도 admin? → true
members.every((m) => m.age >= 20);        // 전부 성인? → false (지호 7살)
```
- `some` = OR(하나라도), `every` = AND(전부). 결과는 **불리언**.
- 전통 웹에서 `let found = false;` 두고 for 돌며 `found = true; break;` 하던 것 = `some` 한 줄.

## 3) `includes` — 값이 **들어있나** (원시값 배열)
```js
const tags = ["식비", "교통", "문화"];
tags.includes("교통"); // true
```
- 객체 배열엔 못 씀(참조 비교라). 객체에서 조건 찾기는 `some`.

## 4) `sort` — 정렬 ⚠️ 함정 주의
```js
const nums = [10, 2, 33, 4];
nums.sort();                  // [10, 2, 33, 4] → "10"<"2"<"33"<"4" 문자열 비교 → [10, 2, 33, 4] 엉뚱
nums.sort((a, b) => a - b);   // [2, 4, 10, 33]  오름차순
nums.sort((a, b) => b - a);   // [33, 10, 4, 2]  내림차순
```
- **함정 1:** 비교함수 없으면 **문자열로 변환해 정렬** → 숫자가 뒤죽박죽. 숫자는 **항상 `(a,b)=>a-b`**.
- **함정 2:** `sort`는 **원본 배열을 직접 바꾼다(mutate).** map/filter와 다름!
  - React state 배열을 직접 sort하면 안 됨 → **복사 후 정렬**: `[...arr].sort(...)` (05장 스프레드).
- 비교함수 규칙: 반환값이 **음수면 a먼저, 양수면 b먼저, 0이면 유지.**

문자열/날짜 정렬:
```js
schedule.sort((a, b) => a.date.localeCompare(b.date)); // "20260622" 같은 문자열 날짜 정렬
```
(날짜 문자열이 `YYYYMMDD` 형태라 사전순 = 시간순이라 편함.)

---

## ⚡ 직접 해보기
1. `members`에서 이름이 `"민지"`인 사람 찾기
2. 미성년자(age < 20)가 **한 명이라도** 있는지 true/false
3. `members`를 **나이 많은 순**으로 정렬한 새 배열 (원본 건드리지 말기 → `[...members]`)
4. (생각) `sort`를 React state에 바로 쓰면 왜 위험할까?

---

## ✅ 체크포인트
- 하나 찾기 `find`(원소)/`findIndex`(위치), 여러 개 `filter`.
- 존재 여부 `some`(하나라도)/`every`(전부)/`includes`(원시값).
- `sort`는 **원본 변경 + 숫자는 비교함수 필수** → state엔 `[...arr].sort()`.

<summary>정답</summary>

```js
// 1
members.find((m) => m.name === "민지");
// 2
members.some((m) => m.age < 20); // true
// 3
[...members].sort((a, b) => b.age - a.age);
// 4: sort는 원본을 직접 바꿔서, state를 직접 수정 → React가 변경 감지 못 하거나
//    불변성 원칙(immutability)을 깨서 버그. 항상 복사본을 정렬해 새 배열을 setState.
```
