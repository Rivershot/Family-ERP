# React 실전 — 가계부로 익히는 핵심 훅·패턴

> Dashboard(조회 전용)에선 `useState`·`useEffect`·`axios`·props·`map`·조건부렌더까지 썼다.
> **가계부**는 *입력 폼 + 거래 목록 CRUD + 집계*라서, 거기서 안 나온 **React 핵심 8할**(폼·불변업데이트·파생상태·커스텀훅·전역상태)이 자연스럽게 쏟아진다.
> **이 트랙은 가계부를 만들며** 그 개념들을 손으로 익히는 게 목표.

## 출발점 (이미 쓰는 것)

`useState` · `useEffect` · `axios.get().then()` · **props 내리기**(차트) · 리스트 렌더(`map`) · 조건부렌더(`link && ...`) · `key` · TS 타입

## 사용법 (js-기초돌파와 동일)

- **읽기만 말고 손으로 친다.** 각 장 `⚡ 직접 해보기`는 답 보기 전에 먼저.
- 개념은 **작은 예제**로 익히고 → 바로 **가계부 화면에 적용**(배운 걸 써먹어야 안 휘발).
- 전통 웹(WebSquare 등) 비유를 군데군데. 막히면 장 번호로 질문.

## 🧠 먼저 잡을 사고 모델 (이거 하나가 React 전부의 뼈대)

```
UI = f(state)
```
- 화면은 **state의 함수**다. state가 바뀌면 React가 그 컴포넌트를 **다시 호출(리렌더)**해 새 UI를 그린다.
- 그래서 **DOM을 직접 만지지 않는다.** (전통 웹: `document.getElementById(...).value = ...`)
  → React: state를 바꾸면 화면이 따라온다. **"무엇을 그릴지"를 선언**하고 "어떻게 바꿀지"는 React에 맡김.
- **불변성**이 변경 감지 장치다. state를 **새 값으로 교체**해야 React가 "바뀌었네" 하고 리렌더. (직접 mutate하면 못 알아챔 → 03강 약점과 같은 뿌리)

> 깊이 원칙(js 트랙과 동일): 각 장은 **"실무에서 읽고 쓸 만큼"** 먼저. 그 아래 **원리**(리렌더·재조정, 클로저로 인한 stale state, 참조 동일성 등)는 ⛏️ 표시한 〔더 파기〕코너로 — 처음엔 건너뛰고 한 바퀴 돈 뒤/막힐 때 깊게.

## 단계별 로드맵 (가계부 기능 순서대로)

### 1단계 — 폼 & 이벤트 (입력 받기) ← **여기부터**
가계부의 시작 = 지출/수입 **입력**. Dashboard엔 입력이 0개였다.
- [ ] 01 **제어 컴포넌트 & 폼** — `value`+`onChange`, `onSubmit`/`preventDefault`, 입력 여러 개를 `name` 기반 한 핸들러로
- [ ] 02 ⛏️ **이벤트 핸들링** — 이벤트 객체(`e.target`), 합성 이벤트, 핸들러에 인자 넘기기(`() => fn(x)`) 〔더 파기: 클로저 때문에 생기는 stale state 함정〕

### 2단계 — 목록 상태 (CRUD)
입력한 거래를 **목록 state**로 쌓고 추가·삭제·수정.
- [ ] 03 ⛏️ **리스트 렌더 & key 심화** — `key`가 왜 중요한가(재조정), **index를 key로 쓰면 안 되는 이유** 〔더 파기: 재조정(reconciliation) 동작〕
- [ ] 04 **불변 업데이트 CRUD** — 추가 `[...list, new]` / 삭제 `filter` / 수정 `map`·`with`. **[[05-구조분해-스프레드-rest]] 직결** ⭐ (03강 불변성 약점 손으로 교정)
- [ ] 05 **파생 상태(derived state)** — 합계·잔액·필터결과는 **state 아님**. 렌더 중 `reduce`/`filter`로 **계산**. "state에 안 넣기" 원칙(초보 최대 함정)

### 3단계 — 구조 (컴포넌트 쪼개기)
한 컴포넌트가 커지면 쪼갠다. *(work.md 2순위 A와 같은 작업)*
- [ ] 06 **컴포넌트 추출 & props** — `<DashboardCard>` 추출, props 타입 설계, **`children`**(합성, slot 같은 것)
- [ ] 07 **state 끌어올리기(lifting state up)** — 형제끼리 데이터 공유, 부모가 state 들고 **콜백을 props로** 내림 ([[props vs state]])

### 4단계 — 커스텀 훅 (로직 재사용) ⭐ 네가 궁금해한 부분
"use로 시작하는 그냥 함수." **컴포넌트가 아니라 상태 *로직*을 재사용**한다.
- [ ] 08 **커스텀 훅 기초** — `useToggle`(모달 열고닫기), `useForm`(폼 value/onChange/reset 묶기). 훅 규칙(최상위에서만 호출)
- [ ] 09 **`useFetch`** — 지금 Dashboard의 **중복 `useEffect`+`axios` 3덩어리를 한 훅으로**. `{ data, loading, error }` 반환 ⭐ (React Query로 가는 다리)
- [ ] 10 **`useLocalStorage`** — state ↔ 브라우저 저장소 동기화. 새로고침해도 가계부 유지

### 5단계 — 성능 & 복잡한 상태
- [ ] 11 ⛏️ **`useMemo` / `useCallback`** — **대부분 불필요**, 언제 진짜 필요한가. 〔더 파기: 참조 동일성·리렌더 비용〕
- [ ] 12 **`useReducer`** — 항목 add/edit/delete를 `dispatch({type})`로. **useState가 복잡해지는 순간의 전환점**(가계부 = 교과서 예제)
- [ ] 13 **`useRef`** — 입력칸 포커스(DOM 접근), 렌더 안 일으키는 값 보관

### 6단계 — 전역 상태 & 라우팅
- [ ] 14 **Context** — prop drilling 해소, 전역 가계부/로그인 사용자. **언제 쓰고 언제 과한가**
- [ ] 15 **Router 심화** — `useParams`(거래 상세), `useSearchParams`(필터를 URL에), 중첩 라우트·탭 연결
- [ ] 16 **(선택) 상태관리 라이브러리 맛보기** — React Query/Zustand가 *어떤 문제*를 풀어주나 (08~14를 손으로 겪은 뒤라야 와닿음)
